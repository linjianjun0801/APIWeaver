说明（我已经把实现要点写清）：

启动方式：在仓库根目录下运行
python -m uvicorn apiweaver.admin_http:app --host 127.0.0.1 --port 9000
注册 API（示例 curl）：
curl -X POST "http://127.0.0.1:9000/admin/register" -H "Content-Type: application/json" -d '{"config": { ... }}'
列表：GET http://127.0.0.1:9000/admin/list
测试：POST http://127.0.0.1:9000/admin/test body: {"api_name":"weather"}
卸载：POST http://127.0.0.1:9000/admin/unregister body: {"api_name":"weather"}
Schema：GET http://127.0.0.1:9000/admin/schema/weather or http://.../schema/weather?endpoint=get_current_weather
实现要点与注意事项（重要）：

我复用并调用了仓库中 APIWeaver 的内部方法（_create_http_client、_create_endpoint_tool）和模型（APIConfig）。这能确保行为与 MCP 工具中的 register/unregister 一致，但属于内部 API（下划线方法），将来库内部改动可能需要同步调整。
持久化采用 apis.json（同进程读写）。若你需要在多实例/多进程间共享，建议使用数据库或 redis，并加分布式锁。
并发/安全：
目前示例没有对 admin 路由做认证，请务必在生产中添加鉴权（API Key、JWT、基本认证、mTLS 等）。
文件 IO 使用简单锁保护（异步锁），但在多进程场景需要更可靠的外部锁或 DB。
与 MCP 服务的关系：
该 admin server 与 APIWeaver 实例同在一个进程中，注册后会调用 weaver.mcp.add_tool(...) 从而让 MCP 工具可用（前提是 MCP server 正在运行并且客户端能看到新增工具）。如果你以 streamable-http transport 启动 MCP，需要确保 MCP 也在运行并且 weaver.mcp.run 在合适的协程/线程中。常见做法是在同一进程启动时把 weaver.run(...) 放到后台任务中（示例里未直接运行 mcp.run，避免阻塞 uvicorn）。
启动时自动加载：ap is.json 中的项会在 FastAPI startup 时被加载并为每个 endpoint 创建对应的工具。
